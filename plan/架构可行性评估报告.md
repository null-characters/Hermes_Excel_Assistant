# Hermes WeCom Excel Assistant 架构可行性评估报告

> **评估日期**: 2026-05-14  
> **评估视角**: 专业架构师  
> **评估对象**: Hermes WeCom Excel Assistant MVP 规划

---

## 一、核心发现

### 1.1 Hermes Agent 能力确认

| 能力维度 | 支持状态 | 备注 |
|----------|----------|------|
| **WeCom 企业微信集成** | ✅ 原生支持 | 内置 `hermes-wecom-callback` 工具集 |
| **文件接收** | ⚠️ 部分支持 | WeCom Callback **不支持文件输入** |
| **文件发送** | ✅ 支持 | 可通过主动 API 发送文件 |
| **Python 代码执行** | ✅ 支持 | `execute_code` 工具 + Docker 后端 |
| **消息网关** | ✅ 支持 | 单一进程管理多平台 |
| **Skills 系统** | ✅ 支持 | 可定义 Excel 处理技能 |
| **Memory 系统** | ✅ 支持 | 持久化用户偏好 |

### 1.2 关键限制

```
⚠️ WeCom Callback 限制（官方文档原文）:
- 不支持流式输出
- 不支持输入指示器
- 仅支持文本输入 — 图片/文件/语音输入尚未实现
- 响应延迟：代理会话需要 3-30 分钟
```

## 二、架构可行性分析

### 2.1 原规划 vs Hermes 实际能力

| 原规划假设 | 实际情况 | 影响 |
|------------|----------|------|
| 需要自建 WeCom Bridge | Hermes 内置 WeCom Callback | ✅ 可简化架构 |
| 通过 HTTP API 调用 Hermes | Hermes 通过消息网关工作 | ⚠️ 需调整集成方式 |
| 用户发送 Excel 文件 | WeCom Callback 不支持文件输入 | ❌ **核心功能受阻** |
| Python Sandbox 独立部署 | Hermes 内置 Docker 后端 | ✅ 可简化架构 |

### 2.2 架构调整建议

```
原规划架构:
┌─────────┐     ┌─────────────┐     ┌───────────────┐     ┌────────────────┐
│ 企业微信  │────▶│ WeCom Bridge │────▶│ Hermes Agent  │────▶│ Python Sandbox │
└─────────┘     └─────────────┘     └───────────────┘     └────────────────┘
                      (自建)              (HTTP API)           (独立容器)

调整后架构:
┌─────────┐     ┌───────────────────────────────────────────────┐
│ 企业微信  │────▶│              Hermes Agent (All-in-One)        │
└─────────┘     │  ┌─────────────┐    ┌────────────────────────┐  │
                │  │ WeCom       │    │ Docker Backend         │  │
                │  │ Callback    │───▶│ (Python + Pandas)      │  │
                │  │ (内置)       │    │ (内置沙箱)              │  │
                │  └─────────────┘    └────────────────────────┘  │
                └───────────────────────────────────────────────┘
```

## 三、可行性结论

### 3.1 核心问题：文件输入不支持

**这是规划的最大障碍。** WeCom Callback 目前仅支持文本输入，无法直接接收用户发送的 Excel 文件。

### 3.2 变通方案

| 方案 | 实现方式 | 用户体验 | 可行性 |
|------|----------|----------|--------|
| **方案 A: 链接方式** | 用户上传文件到网盘/对象存储，发送链接给 Agent | 较差，多一步操作 | ✅ 可行 |
| **方案 B: 企业微信应用菜单** | 通过自定义菜单触发文件上传页面 | 中等，需要 Web 页面 | ✅ 可行 |
| **方案 C: 等待官方更新** | 等待 Hermes 支持 WeCom 文件输入 | 未知 | ⏳ 待定 |
| **方案 D: 混合模式** | WeCom 发送文本指令，通过其他渠道上传文件 | 较差，割裂体验 | ⚠️ 勉强可行 |

### 3.3 推荐方案：方案 B（企业微信应用菜单 + Web 上传）

```
用户流程:
1. 用户在企微应用菜单点击 "上传文件"
2. 打开 Web 上传页面（企微 H5 应用）
3. 上传 Excel 文件，获得文件 ID
4. 返回企微聊天，发送 "@助手 处理文件 xxx"
5. Agent 根据文件 ID 获取文件并处理
6. 处理完成后，Agent 主动发送结果文件
```

## 四、技术实现路径

### 4.1 配置清单（修正版）

```bash
# 环境变量配置
WECOM_CALLBACK_CORP_ID=your-corp-id
WECOM_CALLBACK_CORP_SECRET=your-corp-secret
WECOM_CALLBACK_AGENT_ID=1000002
WECOM_CALLBACK_TOKEN=your-callback-token
WECOM_CALLBACK_ENCODING_AES_KEY=your-43-char-aes-key
WECOM_CALLBACK_PORT=8645

# 终端后端配置（Docker 隔离执行）
TERMINAL_BACKEND=docker
TERMINAL_DOCKER_IMAGE=python:3.11-slim
TERMINAL_CONTAINER_CPU=1
TERMINAL_CONTAINER_MEMORY=2048
```

### 4.2 Docker Compose（简化版）

```yaml
version: '3.8'
services:
  hermes:
    image: nousresearch/hermes-agent:latest
    container_name: hermes-excel-assistant
    ports:
      - "8645:8645"  # WeCom Callback 端口
    volumes:
      - ./data:/app/data
      - ./config:/root/.hermes
    environment:
      - WECOM_CALLBACK_CORP_ID=${WECOM_CALLBACK_CORP_ID}
      - WECOM_CALLBACK_CORP_SECRET=${WECOM_CALLBACK_CORP_SECRET}
      - WECOM_CALLBACK_AGENT_ID=${WECOM_CALLBACK_AGENT_ID}
      - WECOM_CALLBACK_TOKEN=${WECOM_CALLBACK_TOKEN}
      - WECOM_CALLBACK_ENCODING_AES_KEY=${WECOM_CALLBACK_ENCODING_AES_KEY}
      - TERMINAL_BACKEND=docker
      - TERMINAL_DOCKER_IMAGE=python:3.11-slim
    # 注意：需要 Docker-in-Docker 或挂载宿主机 Docker socket
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
```

## 六、结论与建议

### 6.1 总体可行性：**中等偏低**

- **核心功能受阻**：WeCom Callback 不支持文件输入是最大障碍
- **架构可简化**：Hermes 内置能力可替代原规划的多个组件
- **变通方案可行**：通过 Web 上传页面可绕过限制，但用户体验下降

### 6.2 建议

1. **短期（MVP）**：采用方案 B，开发企微 H5 上传页面
2. **中期**：关注 Hermes 更新，等待原生文件输入支持
3. **长期**：考虑贡献代码给 Hermes，实现 WeCom 文件输入功能

### 6.3 是否继续？

**建议继续，但需调整预期和实现方式。** 核心价值（Excel 自动化处理）仍可实现，只是用户交互方式需要调整。
