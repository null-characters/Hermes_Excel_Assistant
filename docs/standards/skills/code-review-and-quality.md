# Code Review and Quality

## 概述

多维度代码审查与质量门禁。**每项更改在合并前都必须经过审查——没有例外。** 审查涵盖五个维度：正确性、可读性、架构、安全性和性能。

**审批标准**：当更改明确改善整体代码健康状况时批准，即使它并不完美。完美的代码不存在——目标是持续改进。不要因为代码不是你写的那种方式就阻止合并。如果它改善了代码库并遵循项目约定，就批准它。

---

## 使用时机

- 合并任何 PR 或更改之前
- 完成功能实现后
- 当另一个代理或模型产出了需要评估的代码时
- 重构现有代码时
- 任何错误修复之后（同时审查修复和回归测试）

---

## 五轴审查

每个审查从以下维度评估代码：

### 1. 正确性

代码是否做到了它声称要做的事情？

- 是否符合规范或任务要求？
- 边界情况是否处理（null、空值、边界值）？
- 错误路径是否处理（不仅仅是正常路径）？
- 是否通过所有测试？测试是否真正测试了正确的事情？
- 是否存在差一错误、竞态条件或状态不一致？

### 2. 可读性与简洁性

另一个工程师（或代理）能否在没有作者解释的情况下理解这段代码？

- 命名是否具有描述性并与项目约定一致？（不要用 `temp`、`data`、`result` 这种没有上下文的名称）
- 控制流是否直观（避免嵌套三元表达式、深层回调）？
- 代码组织是否合理（相关代码分组、清晰的模块边界）？
- 是否有应该简化的"聪明"技巧？
- **能否用更少的行数完成？**（1000行代码能完成的事用了100行是失败）
- **抽象是否值得其复杂性？**（在第三个用例之前不要泛化）
- 注释是否有助于澄清非显而易观的意图？（但不要注释显而易见的代码。）
- 是否存在死代码残留：无操作变量（`_unused`）、向后兼容垫片、或 `// removed` 注释？

### 3. 架构

更改是否符合系统的设计？

- 是否遵循现有模式或引入新模式？如果是新的，是否合理？
- 是否保持清晰的模块边界？
- 是否存在应该共享的代码重复？
- 依赖流向是否正确（没有循环依赖）？
- 抽象级别是否适当（不过度工程化，也不过度耦合）？

### 4. 安全性

更改是否引入漏洞？

- 用户输入是否经过验证和清理？
- 密钥是否远离代码、日志和版本控制？
- 是否在需要的地方检查身份验证/授权？
- SQL 查询是否参数化（没有字符串拼接）？
- 输出是否编码以防止 XSS？
- 依赖是否来自可信来源且没有已知漏洞？
- 来自外部源的数据（API、日志、用户内容、配置文件）是否被视为不可信？
- 外部数据流是否在系统边界验证后才用于逻辑或渲染？

### 5. 性能

更改是否引入性能问题？

- 是否有 N+1 查询模式？
- 是否有无限循环或不受约束的数据获取？
- 是否有应该是异步的同步操作？
- UI 组件是否有不必要的重新渲染？
- 列表端点是否缺少分页？
- 热路径中是否创建了大型对象？

## 变更大小

小型、聚焦的更改更容易审查、更快合并、更安全部署。目标大小：

```
~100 行更改   → 好。一次审查完成。
~300 行更改   → 可接受，如果是单一逻辑更改。
~1000 行更改  → 太大。拆分它。
```

**什么算作"一个更改"**：一个独立的修改，解决一件事，包含相关测试，提交后保持系统功能正常。功能的一部分——不是整个功能。

**更改过大时的拆分策略**：

| 策略 | 方法 | 适用场景 |
|------|------|----------|
| **堆叠** | 提交小更改，基于它开始下一个 | 顺序依赖 |
| **按文件组** | 为需要不同审查者的组分别创建更改 | 横切关注点 |
| **水平** | 先创建共享代码/桩，再创建消费者 | 分层架构 |
| **垂直** | 拆分成更小的全栈功能切片 | 功能开发 |

**将重构与功能工作分离**：一个既重构现有代码又添加新行为的更改是两个更改——分别提交。

## 审查流程

### 步骤 1：理解上下文

在看代码之前，理解意图：
- 这个更改试图完成什么？
- 它实现的是哪个规范或任务？
- 预期的行为变化是什么？

### 步骤 2：先审查测试

测试揭示意图和覆盖范围：
- 更改是否有测试？
- 它们测试的是行为（不是实现细节）吗？
- 边界情况是否覆盖？
- 测试是否有描述性名称？
- 如果代码改变，测试能否捕获回归？

### 步骤 3：审查实现

带着五个维度走查代码：
- 正确性：这段代码是否做了测试说它应该做的？
- 可读性：我能在没有帮助的情况下理解它吗？
- 架构：这是否符合系统？
- 安全性：有任何漏洞吗？
- 性能：有任何瓶颈吗？

### 步骤 4：分类发现

为每条评论标注严重程度：

| 前缀 | 含义 | 作者行动 |
|------|------|----------|
| *(无前缀)* | 必需更改 | 合并前必须解决 |
| **Critical:** | 阻止合并 | 安全漏洞、数据丢失、功能损坏 |
| **Nit:** | 次要、可选 | 作者可忽略——格式、风格偏好 |
| **Optional:** | 建议 | 值得考虑但非必需 |
| **FYI** | 仅供参考 | 无需行动 |

## 死代码清理

任何重构或实现更改后，检查孤立代码：

1. 识别现在不可达或未使用的代码
2. 明确列出
3. **删除前询问**："我应该删除这些现在未使用的元素：[列表]？"

不要留下死代码——它会混淆未来的读者和代理。但不要默默删除你不确定的东西。

---

## 诚实审查

审查代码时——无论是你、另一个代理还是人类编写的：

- **不要橡皮图章**：没有审查证据的 "LGTM" 对谁都没帮助。
- **不要弱化真实问题**：当它是会在生产环境中出现的 bug 时说"这可能是一个小问题"是不诚实的。
- **尽可能量化问题**："这个 N+1 查询将为列表中的每个项目增加约 50ms" 比 "这可能很慢" 更好。
- **对有明显问题的方法提出反对**：如果实现有问题，直接说出来并提出替代方案。
- **优雅地接受否决**：如果作者有完整上下文并不同意，尊重他们的判断。

## 审查清单

```markdown
## 审查：[PR/更改标题]

### 正确性
- [ ] 更改符合规范/任务要求
- [ ] 边界情况已处理
- [ ] 错误路径已处理
- [ ] 测试充分覆盖更改

### 可读性
- [ ] 命名清晰一致
- [ ] 逻辑直观
- [ ] 无不必要的复杂性

### 架构
- [ ] 遵循现有模式
- [ ] 无不必要的耦合或依赖
- [ ] 适当的抽象级别

### 安全性
- [ ] 代码中无密钥
- [ ] 边界处输入已验证
- [ ] 无注入漏洞
- [ ] 认证检查到位

### 性能
- [ ] 无 N+1 模式
- [ ] 无无限操作
- [ ] 列表端点有分页

### 验证
- [ ] 测试通过
- [ ] 构建成功
- [ ] 手动验证已完成（如适用）

### 结论
- [ ] **批准** — 准备合并
- [ ] **请求更改** — 问题必须解决
```

## 红旗警告

- 没有任何审查就合并的 PR
- 只检查测试是否通过的审查（忽略其他维度）
- 没有实际审查证据的 "LGTM"
- 没有安全审查的安全敏感更改
- "太大无法正确审查"的大型 PR（拆分它们）
- 错误修复 PR 没有回归测试
- 没有严重程度标签的审查评论
- 接受"我以后会修复"——这永远不会发生

## 常见借口

| 借口 | 现实 |
|------|------|
| "它能工作，这就够了" | 能工作但不可读、不安全或架构错误的代码会产生复利债务。 |
| "我写的，所以我知道它是正确的" | 作者对自己的假设视而不见。每个更改都受益于另一双眼睛。 |
| "我们以后会清理" | 以后永远不会来。审查是质量门禁——使用它。 |
| "AI 生成的代码可能没问题" | AI 代码需要更多审查，而不是更少。它自信且看似合理，即使错误时也是如此。 |
| "测试通过了，所以没问题" | 测试是必要但不充分的。它们不能捕获架构问题、安全问题或可读性问题。 |
