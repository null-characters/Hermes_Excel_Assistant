# Karpathy Guidelines

## 概述

行为准则，用于减少常见的 LLM 编码错误，源自 Andrej Karpathy 的观察。这些准则对抗三个核心 LLM 编码陷阱：做出错误假设并在未检查的情况下执行、用臃肿的抽象过度复杂化代码、以及对未完全理解的代码/注释进行意外修改。

> **权衡说明**：这些准则倾向于谨慎而非速度。对于简单任务（明显的一行代码、琐碎的修复），请自行判断——并非每个更改都需要完整的严格流程。

---

## 1. 先思考后编码

**不要假设。不要隐藏困惑。揭示权衡。**

在实现之前：
- 明确陈述假设。如果不确定，先询问。
- 如果存在多种理解，呈现所有选项——不要默默选择其中一个。
- 如果存在更简单的方法，说出来。必要时提出反对意见。
- 如果有不清楚的地方，停下来。指出困惑。提问。

---

## 2. 简洁优先

**用最少的代码解决问题。不做推测性设计。**

- 不要添加超出请求的功能
- 不要为单次使用的代码创建抽象
- 不要添加未请求的"灵活性"或"可配置性"
- 不要为不会发生的场景编写错误处理
- 如果 50 行能解决问题但写了 200 行——重写它

**自检**："资深工程师会认为这过度复杂吗？"如果是，简化。

---

## 3. 手术式变更

**只改必须改的。只清理自己制造的混乱。**

编辑现有代码时：
- 不要"改进"相邻代码、注释或格式
- 不要重构没有坏的东西
- 匹配现有风格，即使你会用不同方式
- 如果注意到无关的死代码，提及它——不要删除

当你的变更产生孤立代码时：
- 删除因你的变更而变得未使用的导入/变量/函数
- 不要删除先前存在的死代码，除非被要求

**试金石**：每一行变更都应能直接追溯到用户的请求。

---

## 4. 目标驱动执行

**定义成功标准。循环直到验证。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写复现 bug 的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

**强成功标准**使独立迭代成为可能。**弱标准**（"让它工作"）需要持续澄清。

---

## 验证

这些准则生效的标志：
- diff 中出现更少的不必要更改
- 因过度复杂性导致的重写减少
- 澄清问题在实现之前提出，而非在错误之后
- 干净、最小化的 PR
