# 通用工作流规范（完整五阶段）

项目无关的标准工作流：需求澄清 → 规格输出 → 任务拆解 → 执行落地 → 校验评估。阶段二、三的产出可视为**规格驱动（SDD）**的「图纸与验收边界」；阶段四对代码任务可结合**测试驱动（TDD）**落地，见下文「规格与测试的衔接」。

## 路径抉择（强制，先于阶段一）

用户要求 **workflow / 工作流** 或等价表述时：**在进入阶段一之前**，必须先读 `docs/standards/workflow.md` 的 **「调用方式」** 与 **「轻量级变更（分流）」**，**显式输出**所选路径（A 完整五阶段 / B 轻量级）及一行理由。

- **仅当**抉择为 **路径 A** 或用户明确要求完整 workitem 时，继续执行本 skill 下文五阶段。
- 若抉择为 **路径 B**，应改用 **lightweight-change**，不得用本 skill 代替短流程。

## 目录结构约定

从项目根目录起，相对结构为：

```
<项目根目录>/
├── CODEBUDDY.md                    # 项目入口配置（可选）
└── docs/
    ├── workitems/                  # 任务项目：每个任务独立文件夹
    │   └── <任务名>/
    │       ├── plan.md             # 规划文档
    │       ├── tasks.md            # 任务清单
    │       └── review.md           # 校验评估报告
    ├── standards/                  # 规范文档
    │   └── *.md
    └── debug/                      # 调试记录
        └── *.md
```

原则：每个任务独立文件夹，含 plan / tasks / review；任务名用中文文件夹，内部文件用英文命名。

## 五阶段工作流

### 阶段一：需求澄清

**触发**：用户提出新需求或问题。

**步骤**：
1. 明确目标范围
2. 确认当前基线（已有实现、约束）
3. 明确交付目标（文档、代码、方案）
4. 若信息不足，提出 3～5 个核心问题

**输出**：澄清结论（口头或写入规划文档）。

---

### 阶段二：规格输出

**触发**：需求澄清完成。

**步骤**：
1. 在 `docs/workitems/<任务名>/` 创建文件夹
2. 创建 `plan.md`，含版本历史、背景与目标、当前基线、方案设计、风险与避坑、验收标准

完整 plan 模板见下文「plan.md 模板」。

---

### 阶段三：任务拆解

**触发**：规划文档确认。

**步骤**：
1. 在 `docs/workitems/<任务名>/` 创建 `tasks.md`
2. 按 P0（核心）→ P1（重要）→ P2（优化）拆解原子任务，表格含：编号、任务描述、影响模块、依赖、状态；**建议**为代码类任务增加可选列「测试要点」（TDD 范围或验收级测试说明）

**状态**：`待开始` | `进行中` | `已完成` | `已阻塞`

完整 tasks 模板见下文「tasks.md 模板」。

---

### 阶段四：执行落地

**触发**：用户明确要求执行任务。

**规则**：
- 文档任务：直接修改对应文档
- 代码任务：先识别影响范围，再执行；**必须遵循 [Karpathy Guidelines](./karpathy-guidelines.md)**
- **禁止未经用户确认自动提交版本库**

**顺序**：P0 → P1 → P2；有依赖按依赖顺序，无依赖可并行。

**规格与测试的衔接（SDD / TDD）**：

- **阶段二 `plan.md`**：验收标准宜写成**可判定、可验证**的条件（便于阶段五对照，并向下游分解为测试意图）。
- **阶段三 `tasks.md`**：原子任务与 `plan` 验收对齐；代码类任务可在「测试要点」列标明本任务对应的 TDD 范围或须覆盖的验收点。
- **阶段四（代码任务）**：**建议**采用 TDD（先写失败测试 → 实现 → 重构），使实现能追溯至 `plan.md` 与 `tasks.md` 对应条目。纯文档、纯配置、或用户另有约定的任务，不必强行 TDD。
- 若在 TDD/实现中发现**规格不可行、不可测或与成本严重不符**，应先修订 `plan.md` / `tasks.md`（含版本历史）再改代码，避免实现静默偏离「图纸」。

**Cursor 多 agent**：子会话派工、SSOT 与并行边界见下文「Cursor 多 agent 派工约定」。

---

### 阶段五：校验评估

**触发**：任务执行完成或用户要求校验/提交前。

**步骤**：
1. 统计任务完成度
2. 汇总代码变更清单（新增/修改/删除）
3. 生成函数/类级别审查清单
4. 评估副作用风险
5. 在 `docs/workitems/<任务名>/` 输出 `review.md`

**代码审查**：必须遵循 [Code Review and Quality](./code-review-and-quality.md) 五轴审查框架（正确性、可读性、架构、安全性、性能）

完整 review 模板见下文「review.md 模板」。

---

## 铁律

1. **执行 workflow 须先做路径抉择**（见上文「路径抉择」与 `docs/standards/workflow.md`「调用方式」）。
2. **禁止在未完成需求澄清前直接改动代码**
3. **禁止未经用户明确要求时自动提交版本库**
4. **每个任务的 plan / tasks / review 必须在同一文件夹内维护**
5. **每次变更必须更新文档版本历史**
6. **校验评估未通过时禁止提交**
7. **规格与实现冲突时先改文档**：若验收标准、任务拆分与可测性或实现发现矛盾，须先更新 `plan.md` / `tasks.md`（及版本历史），再调整代码与测试，避免无文档偏离
8. **代码编写遵循 Karpathy Guidelines**：手术式变更、简洁优先、目标驱动；详见 [karpathy-guidelines](./karpathy-guidelines.md)
9. **代码审查遵循五轴框架**：正确性、可读性、架构、安全性、性能；审查评论必须标注严重程度；详见 [code-review-and-quality](./code-review-and-quality.md)
10. **禁止以文件太大为由精简内容**：若文件写入失败，必须分段多次写入，不得以"精简""简写"等理由删减用户要求的内容

## Cursor 多 agent 派工约定

将 **Cursor 多会话 / 子 agent** 与五阶段工作流对齐的编排规则（不改变阶段定义与铁律）。

### 角色分工

| 角色 | 典型承载 | 职责 |
|------|-----------|------|
| 编排器 | 主会话 | 阶段一～三：`plan.md`、`tasks.md`；阶段四：按依赖派发与收口；阶段五：汇总或发起评审 |
| 执行器 | 子 agent / 独立会话 | 认领明确工作包：路径边界、`tasks.md` 的 `T-xx`、`plan.md` 验收与禁则 |
| 评审（可选） | 只读子 agent | 阶段五：对照文档与变更填写/核对 `review.md`，不擅自改业务代码 |

### 单一事实来源（SSOT）

- **范围与验收**：`docs/workitems/<任务名>/plan.md`。
- **任务与依赖**：同目录 `tasks.md` 的编号、依赖列、状态。
- 派工须写明任务目录、`T-xx`、允许/禁止修改的路径；子 agent 不依赖他会话聊天记录。

### 派发原则

1. 依赖未「已完成」的 `T-xx` 不派工、不并行。
2. 无依赖且路径不冲突的可并行；可能改同一文件则串行或合并任务。
3. 派发/收口时更新 `tasks.md` 状态与版本历史。
4. 任务粒度须可独立验收，避免多代理拆改同一逻辑无锁。

### 派工指令建议携带

任务目录、`T-xx`、路径范围、`plan.md` 相关验收引用、该 `T-xx` 在 `tasks.md` 的「测试要点」列（若已填写）、构建/测试与提交禁则（默认：**禁止未经用户明确要求提交版本库**）。

### 阶段五

执行收口后再校验；评审向子 agent 优先只读，产出进 `review.md`。

## plan.md 模板

```markdown
# <任务名> 规划

## 版本历史
| 版本 | 日期 | 变更说明 |
|------|------|----------|
| V1.0 | YYYY-MM-DD | 初稿 |

## 一、背景与目标
<!-- 描述需求背景、业务价值、目标范围 -->

## 二、当前基线
<!-- 描述已有实现、约束条件、技术债务 -->

## 三、方案设计
<!-- 核心设计内容 -->

## 四、风险点与避坑说明
<!-- 已知风险、边界条件、注意事项 -->

## 五、验收标准
<!-- 可量化、可判定的验收条件；宜便于分解为验收/集成测试意图与单元测试边界 -->
<!-- 可区分：用户可见结果、接口契约、兼容性、非功能（性能/安全）等 -->
```

## tasks.md 模板

```markdown
# <任务名> 任务清单

## 版本历史
| 版本 | 日期 | 变更说明 |
|------|------|----------|
| V1.0 | YYYY-MM-DD | 初稿 |

## 任务列表

### P0 - 核心任务
| 编号 | 任务描述 | 影响模块 | 依赖 | 测试要点（可选） | 状态 |
|------|----------|----------|------|------------------|------|
| T-01 | ... | ... | - | 对应 plan §五 条目 x；TDD：… | 待开始 |

### P1 - 重要任务
| 编号 | 任务描述 | 影响模块 | 依赖 | 测试要点（可选） | 状态 |
|------|----------|----------|------|------------------|------|
| T-02 | ... | ... | T-01 | … | 待开始 |

### P2 - 优化任务
| 编号 | 任务描述 | 影响模块 | 依赖 | 测试要点（可选） | 状态 |
|------|----------|----------|------|------------------|------|
| T-03 | ... | ... | T-02 | … | 待开始 |

## 任务编排原则
1. ...
2. ...
```

**状态定义**：`待开始` | `进行中` | `已完成` | `已阻塞`

**「测试要点」列**：代码类任务建议填写——对应 `plan.md` 验收条款、TDD 范围，或须补的集成/验收级验证；非代码任务可留空或写「N/A」。

## 与项目配置的关系

项目根目录的 `CODEBUDDY.md`（若存在）可定义项目名称、技术栈、关键目录、约束。本工作流通过相对路径引用，与项目配置解耦，可跨项目复用。
