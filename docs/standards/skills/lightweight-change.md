# 轻量级变更流程

面向**小范围 Bug 修复、单模块小需求**：短闭环，**不**要求 `docs/workitems/<任务名>/` 下的 `plan.md` / `tasks.md` / `review.md`。

## 路径抉择（强制，先于下文四步）

- 若用户仅说「workflow / 工作流」而未指定短流程：必须先读 **[docs/standards/workflow.md](../workflow.md)** 的 **「调用方式」** 与 **「轻量级变更（分流）」**，**显式输出**路径 A 或 B 及理由；**仅当抉择为路径 B**（或用户明确要求轻量）时，才执行本 skill 的四步。
- 若抉择为 **路径 A**，须改用 **standard-workflow**，不得继续本 skill。

完整五阶段与分流条件以该文件为准；本 skill 为路径 B 的执行摘要。

---

## 四步流程

### 1. 最小澄清

- 确认：**现象**、**期望行为**、**范围**（平台/模块/入口）。
- 信息不足时先问 **1～3 个**关键问题，再动手。

### 2. 定位与方案

- 给出根因假设或验证路径；方案用**一句话**可验收的描述。
- 若在定位中发现 **跨多模块、协议/存储边界、安全/合规**，**停止**并建议改用 **standard-workflow**（完整 workitem）。

### 3. 实现与自证

- **最小改动集**；优先与现有代码风格一致。
- 按项目习惯做可自动化验证（编译、单测、复现步骤）；无法执行时列出**待用户确认**项。

### 4. 收口

- **不要求**更新 `tasks.md`。
- **可选**：在 `docs/debug/` 或用户指定路径追加极短记录，模板见下文「轻量记录模板」。
- 若需审计追溯或书面验收，**升格**：新建 `docs/workitems/<任务名>/`，走 **standard-workflow**。

---

## 升格条件（满足任一条则改用完整工作流）

与 `docs/standards/workflow.md` 所列保持一致：

- 涉及**多个模块/子系统**或**多人协作**依赖明确排期。
- 需要书面 **plan / tasks / review** 或正式验收、审计留痕。
- **协议、存储格式、持久化兼容**或对外行为契约变更。
- **安全、隐私、合规**相关。
- **无稳定复现**或根因仍大幅不确定。
- 预计改动 **> 3 个文件**（非机械重命名）或 **> 1 个工作日**仍看不清边界。

---

## 铁律（轻量流程不豁免）

1. **用户说 workflow / 工作流时**：须先完成路径抉择（见上文）；未判定为路径 B 不得用本 skill 代替完整流程。
2. **未完成澄清前**不做大范围重构或无关改动。
3. **禁止未经用户明确要求**自动提交版本库（SVN/Git 等）。
4. 校验/提交前若团队要求 **review.md**，应升格为完整工作流或按项目规范补文档。

## 轻量记录模板

可复制到 `docs/debug/`、workitem 备注或用户指定路径；非强制。

```markdown
# 轻量记录：<一句话标题>

| 字段 | 内容 |
|------|------|
| 日期 | YYYY-MM-DD |
| 类型 | Bug 修复 / 小需求 |

## 现象
<!-- 用户可见问题或触发方式 -->

## 根因
<!-- 简述；若仍不确定写「待验证」并升格流程 -->

## 改动
<!-- 文件路径列表或要点 -->

## 验证
<!-- 编译/单测/手工步骤；未做则写待办 -->

## 是否需升格 workitem
- [ ] 否
- [ ] 是（原因：…）
```
