## 1.1 Compose type

1. The program block should be written in indent style, and the number of indented spaces is 4. Note: there can be inconsistencies in the code automatically generated by the development tool.
2. Blank lines must be added between relatively independent program blocks and after variable descriptions.
3. Long sentences (> 80 characters) should be written in multiple lines, long expressions should be divided into new lines at low priority operators, operators should be placed at the beginning of new lines, and the new lines should be indented properly to make the layout neat and the sentences readable.
4. If there is a long expression or statement in the loop, judgment and other statements, it should be divided appropriately. The long expression should be divided into new lines at the low priority operator, and the operator should be placed at the beginning of the new line.
5. If the parameters of a function or procedure are long, it should be divided appropriately.
6. It is not allowed to write multiple phrasal sentences in one line, that is, only one sentence in one line.
7. If, for, do, while, case, switch, default and other statements occupy one line by themselves, and the execution part of if, for, do, while and other statements, no matter how much, should be bracketed {}.
8. The beginning of function or procedure, the definition of structure and the code in the statement of loop and judgment should adopt the indent style, and the case statement should also comply with the statement indent requirements.
9. The separators of program blocks (such as braces' {' and '} 'in C language) should be on the same line and in the same column, and should be left aligned with the statements that refer to them. In the beginning of function body, definition of class, definition of structure, definition of enumeration and if, for, do, while, switch, case statements, the program should adopt the above indentation method.
10. When two or more keywords, variables and constants are operated in the same way, spaces should be added before, after or before the operators between them; When performing non equivalent operations, if it is a closely related immediate operator (such as - >), no spaces should be added after it.

## 1.2 Annotation

1. Generally, the effective annotation amount of source program must be more than 20%.
2. The function header should be annotated to list the function purpose / function, input parameters, output parameters, return values, etc.
3. Write code while commenting, modify the code and modify the corresponding comments to ensure the consistency of comments and code. Comments that are no longer useful should be removed.
4. The content of notes should be clear, clear, and the meaning should be accurate to prevent the ambiguity of notes. Wrong annotation is not only useless but also harmful.
5. Avoid using abbreviations in comments, especially when they are very common.

## 1.3 Identifier rules

1. The name of the identifier should be clear, clear and have a clear meaning. At the same time, complete words or abbreviations that everyone can understand should be used to avoid misunderstanding.
2. If special conventions or abbreviations are used in naming, there should be notes.
3. Their own unique naming style should be consistent from beginning to end and should not be changed back and forth.
4. For variable naming, it is forbidden to take a single character (such as I, J, K...). It is suggested that in addition to having a specific meaning, it can also indicate its variable type, data type, etc., but I, J, K are allowed to be local loop variables.
5. The naming conventions must be consistent with the style of the system used and be unified in the same project.
6. Do not use numbers or strange characters to define identifiers unless necessary.
7. Name mutually exclusive variables or functions with opposite actions with correct antonyms.

## 1.4 Readability

1. Pay attention to the priority of operators, and use parentheses to clarify the operation order of expressions, so as to avoid using the default priority.
2. Avoid numbers that are not easy to understand and replace them with meaningful signs. For constants involving physical state or containing physical meaning, numbers should not be used directly, but meaningful enumeration or macro should be used instead.
3. The closely related codes in the source program should be adjacent as much as possible.
4. Don't use difficult and technical sentences unless it is necessary. High skill statement is not equal to high efficiency program. In fact, the key of program efficiency lies in algorithm.

## 1.5 Variables & Struct

1. Remove unnecessary public variables.
2. Carefully define and clarify the meaning, function, range of values and the relationship between public variables.
3. Clarify the relationship between public variable and function or procedure that operates the public variable, such as access, modification and creation.
4. When transferring data to public variables, we should be careful to prevent unreasonable values or cross boundary phenomena.
5. Prevent local variables from having the same name as public variables.
6. Uninitialized variables are strictly prohibited as the right value.
7. Carefully design the layout and arrangement order of elements in the structure, so that the structure can be easily understood, space consumption can be saved and misuse can be reduced.
8. The design of the structure should consider forward compatibility and future version upgrading as much as possible, and reserve space for some possible future applications (such as some space reserved).
9. Pay attention to the principles and details of specific language and compiler handling different data types.
10. When programming, pay attention to the forced conversion of data types.
11. We should also have a good understanding of the default data type conversion of the compilation system.
12. Minimize unnecessary data type default and cast.
13. Design data reasonably and use custom data types to avoid unnecessary type conversion between data.
14. When declaring data structure for distributed environment or communication environment between different CPU, the byte order of machine, bit field used and byte alignment must be considered.

## 1.6 Function

1. The error return code of the called function should be handled carefully and comprehensively.
2. The function function is clear, and the function design is realized accurately (not approximately).
3. Prevent the parameters of a function from being used as working variables.
4. The size of the function should be limited to 200 lines.
5. A function performs only one function.
6. Write functions for simple functions.
7. Don't design multi-purpose, all inclusive functions. Functions with multiple functions may make it difficult to understand, test and maintain functions.
8. The function should be predictable, that is, as long as the input data is the same, it should produce the same output.
9. Try not to write functions that depend on the internal implementation of other functions.
10. Avoid designing multi parameter functions, and remove unused parameters from the interface.
11. Non scheduling functions should reduce or prevent control parameters, and only use data parameters as far as possible.
12. Check the validity of all parameter inputs to the function.
13. Check the validity of all nonparametric inputs to the function, such as data files, public variables, etc.
14. The function name should accurately describe the function.

## 1.7 Efficiency of programming

1. When programming, we should always pay attention to the efficiency of the code. Code efficiency is divided into global efficiency, local efficiency, time efficiency and space efficiency. Global efficiency is the efficiency of the whole system; Local efficiency is the efficiency from the perspective of module or function; Time efficiency is the time required for a program to process input tasks; Space efficiency is the memory space required by the program, such as machine code space size, data space size, stack space size, etc.
2. On the premise of ensuring the correctness, stability, readability and testability of the software system, improve the code efficiency. We can't blindly pursue code efficiency, which will affect the correctness, stability, readability and testability of software.
3. The local efficiency should serve the global efficiency and should not affect the global efficiency because of improving the local efficiency.
4. The space efficiency is improved by the division and organization of the system data structure and the optimization of the program algorithm.
5. The workload in circulatory system was minimized. We should carefully consider whether the statements in the loop body can be placed outside the loop body to minimize the workload in the loop body, so as to improve the time efficiency of the program.
6. In multiple loops, the busiest loop should be placed in the innermost layer.
7. Minimize loop nesting levels.
8. The loop should be placed in the code block of the judgment statement instead of the judgment statement in the loop.
